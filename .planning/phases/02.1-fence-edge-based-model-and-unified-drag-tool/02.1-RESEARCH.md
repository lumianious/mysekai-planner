# Phase 02.1: Fence Edge-Based Model and Unified Drag Tool — Research

**Researched:** 2026-04-11
**Domain:** Grid editor — data model refactor + interaction unification (React/Konva/Zustand)
**Confidence:** HIGH (fully internal codebase; all facts verified against current source)

## Summary

Phase 2 shipped fences as **tile-based** items placed via a three-phase click-line tool (`idle → picking-end → confirming`) with an inline HTML confirm overlay. Human verification revealed two architectural mistakes: (1) fences in MySekai live on **grid edges** (the lines between tiles), not inside tile cells, so they must coexist with roads/color-tiles on the same (x,y) square; (2) the interaction should match roads/color-tiles — **hold-drag paint**, not click-line-confirm.

This phase re-architects the fence data model from `PlacedItem{layer:'ground'}` to a new edge entity `PlacedEdge{x,y,orientation:'h'|'v'}`, deletes the entire Phase 2 fence line-tool surface (`useFenceLineTool.ts`, `FenceLineTool.tsx`, `FenceConfirmOverlay.tsx`, `fenceLineTool.test.ts`), and folds fence painting into the existing `handleStageMouseDown`/`handleMouseMove` drag-paint path in `EditorCanvas.tsx`. The drag handler branches on `getBrushInteraction(fixture)` (already `'drag-paint'` vs `'line-tool'`) — for fences, the branch changes from "open line-tool state machine" to "start an edge-paint stroke, axis-locked to the dominant axis of the drag."

**Primary recommendation:** Introduce a second occupancy primitive (`placedEdges: Record<string, PlacedEdge>` plus `buildEdgeOccupancyGrid`) parallel to the existing `placedItems`/`buildOccupancyGrid`, keep `getBrushInteraction` returning `'line-tool'` for fences but rename that constant to `'drag-paint-edge'` so the routing layer switches on "writes to edges" vs "writes to tiles," and handle axis-lock inside the drag-paint mouseMove branch by capturing `dragStartAxis` on first cross-tile sample.

## Edge-Based Data Model Design

### Canonical representation

```typescript
// src/types/editor.ts — add after PlacedItem
export type EdgeOrientation = 'h' | 'v'

export interface PlacedEdge {
  id: string                    // UUID, parallel to PlacedItem.id
  fixtureId: number             // which fence variant
  x: number                     // lattice x — see coordinate semantics below
  y: number                     // lattice y
  orientation: EdgeOrientation  // 'h' = horizontal edge (extends in +x direction)
                                // 'v' = vertical edge (extends in +y direction)
}
```

### Coordinate semantics (critical — nail this in the plan)

A grid of `W × D` tiles has `(W+1) × (D+1)` lattice points. For edge `{x, y, orientation}`:

| orientation | Visual meaning | Connects tiles | Valid range |
|-------------|----------------|----------------|-------------|
| `'h'` (horizontal) | Runs along the **top edge of tile (x,y)**, between tiles (x, y-1) above and (x, y) below | (x, y-1) ↔ (x, y) | `0 ≤ x < W`, `0 ≤ y ≤ D` |
| `'v'` (vertical) | Runs along the **left edge of tile (x,y)**, between tiles (x-1, y) left and (x, y) right | (x-1, y) ↔ (x, y) | `0 ≤ x ≤ W`, `0 ≤ y < D` |

Rationale for anchoring to the "top-left of a tile" origin: it mirrors the existing `PlacedItem.x/y = top-left tile` convention in Phase 1, so rendering and hit-testing stay consistent. A horizontal edge at `(x, y)` renders as a thin rectangle at pixel `(x * TILE_SIZE, y * TILE_SIZE - thickness/2)` with width `TILE_SIZE`.

### Edge occupancy

```typescript
// Key convention: `${x},${y},${orientation}`
export type EdgeOccupancySet = Set<string>

export function edgeKey(x: number, y: number, o: EdgeOrientation): string {
  return `${x},${y},${o}`
}

export function buildEdgeOccupancySet(edges: Record<string, PlacedEdge>): EdgeOccupancySet {
  const set = new Set<string>()
  for (const e of Object.values(edges)) set.add(edgeKey(e.x, e.y, e.orientation))
  return set
}

export function checkCanPlaceEdge(
  set: EdgeOccupancySet,
  x: number, y: number, o: EdgeOrientation,
  gridW: number, gridD: number,
): boolean {
  // Bounds per orientation
  if (o === 'h' && (x < 0 || x >= gridW || y < 0 || y > gridD)) return false
  if (o === 'v' && (x < 0 || x > gridW || y < 0 || y >= gridD)) return false
  return !set.has(edgeKey(x, y, o))
}
```

Edge occupancy is a **flat Set**, not a Map to item-id. Unlike tile occupancy (where we need the owner ID to handle overwrite), fence edges are all equivalent to v1 — we only need existence checking. If we later need per-edge variant tracking, the Set becomes a Map to edge ID.

**Why a Set not a Map to `PlacedEdge.id`**: Faster, simpler, no stale-ID cleanup. For erase-mode we iterate `Object.values(placedEdges)` directly (same pattern as `eraseGroundTileIfAny` at EditorCanvas L241). Fence edge counts in a typical blueprint are ≤ 200 — linear iteration is fine.

### Phase 3 URL encoding compatibility

Phase 3 hasn't shipped, so there is no encoded format to break — but the plan must specify the serialized shape so Phase 3 can consume it. Recommended persistence shape:

```typescript
interface SerializedBlueprint {
  version: 1
  areaLevel: AreaLevel
  items: Array<[fixtureId, x, y, rotation, layer]>   // existing PlacedItem compact form
  edges: Array<[fixtureId, x, y, orientationByte]>    // new — orientationByte: 0=h, 1=v
}
```

Orientation is encoded as a single byte (0/1) to minimize URL-hash length. `lz-string` compression downstream will handle repetition efficiently. The `edges` field is **additive** — a Phase 3 implementation that doesn't know about edges (unlikely, but defensive) would still decode `items` correctly. Phase 3 plan must add an `edges` decoder path; this phase should emit a JSDoc note in `types/editor.ts` flagging that change.

**Do not share one array for items + edges.** They have different schemas; combining them forces a discriminator field that wastes bytes.

## Unified Brush/Drag Handler Architecture

### Current drag-paint flow (Phase 2, post-02-03)

```
onMouseDown (Stage, empty area)
  └─ toolMode === 'brush' && activeFixtureId && getBrushInteraction === 'drag-paint'
      ├─ pointerToGrid(stage, step) → {gridX, gridY}
      ├─ startStrokeBatch()
      ├─ isPaintingRef.current = true
      ├─ lastPaintedTileRef.current = {x: gridX, y: gridY}
      └─ paintTileIfAllowed(gridX, gridY, fixture)

onMouseMove
  └─ isPaintingRef.current
      ├─ compute snappedX/Y from cursor
      ├─ rasterizeLine(last, snapped, step) for interpolation
      └─ paintTileIfAllowed for each interpolated tile

onMouseUp / onMouseLeave / window.blur
  └─ endStrokeIfActive() → endStrokeBatch() (zundo commit)
```

### Target unified flow (Phase 02.1)

Two options evaluated:

#### Option A: Discriminated target in the existing handlers (RECOMMENDED)

Keep the same handlers. The `getBrushInteraction(fixture)` result splits into two code paths at the **innermost commit step**:

```typescript
// In handleStageMouseDown, replace the 'drag-paint' branch with:
if (interaction === 'drag-paint' || interaction === 'drag-paint-edge') {
  startStrokeBatch()
  if (interaction === 'drag-paint') {
    // existing tile path — unchanged
    isPaintingRef.current = true
    lastPaintedTileRef.current = { x: p.gridX, y: p.gridY }
    paintTileIfAllowed(p.gridX, p.gridY, fixture)
  } else {
    // new edge path
    isEdgePaintingRef.current = true
    edgeStrokeStartRef.current = { x: p.rawGridX, y: p.rawGridY }
    lastEdgePointRef.current = { x: p.rawGridX, y: p.rawGridY }
    edgeStrokeAxisRef.current = null  // determined at first cross-tile sample
    // no initial edge committed — single-tile click commits 0 edges (user just clicked)
  }
  return
}
```

**Pros:** minimal diff vs current code; reuses `startStrokeBatch`/`endStrokeBatch`; existing mouseleave/blur R-06 cleanup via `endStrokeIfActive` extends naturally.
**Cons:** two parallel ref sets and two branches in mouseMove. Fine at this scale (~30 lines added).

#### Option B: Strategy object per fixture type

```typescript
interface BrushStrategy {
  onStrokeStart(p: PointerTile): void
  onStrokeContinue(p: PointerTile): void
  onStrokeEnd(): void
}
const tileStrategy: BrushStrategy = { ... }
const edgeStrategy: BrushStrategy = { ... }
// Dispatch: const strategy = getBrushInteraction(fixture) === 'drag-paint-edge' ? edgeStrategy : tileStrategy
```

**Pros:** cleaner separation, extensible for v2 brush types.
**Cons:** adds abstraction for two concrete strategies; closures over refs make it hard to colocate with EditorCanvas state; premature abstraction per CLAUDE.md simplicity-first rule.

**Recommendation: Option A.** Two concrete branches are simpler than an interface with two implementers. Revisit strategy pattern only if a third brush target appears in v2.

## Axis-Lock Algorithm

### Core problem statement

User mouses down at tile A, drags through tiles B, C, D. Determine whether the stroke lays **horizontal edges** (constant y, varying x) or **vertical edges** (constant x, varying y). Diagonal input must collapse to one axis.

### Three candidate strategies

| Strategy | Logic | Pros | Cons |
|----------|-------|------|------|
| **Lock at first cross-tile sample** | Axis = `|Δx| > |Δy| ? 'h' : 'v'` measured at the first mouseMove sample where `rawGridX ≠ startX` or `rawGridY ≠ startY`. Once locked, orthogonal motion is ignored for the rest of the stroke. | Predictable; matches Paper/Figma constrained-line behavior; no visual flicker | If user starts horizontal then wants vertical, they must release and restart |
| **Rebase on dominant delta every sample** | Each mouseMove, recompute axis from `|currentX - startX|` vs `|currentY - startY|`. Preview updates. | User can "correct" mid-drag | Confusing: mid-drag axis flip wipes previously-laid edges; needs preview vs commit distinction |
| **Lock at mouseMove #1 using total delta** | Wait for first sample where `|Δx| + |Δy| ≥ 1`, then lock. Same as #1 really. | Identical to #1 | Identical to #1 |

**Recommendation: Lock at first cross-tile sample (Strategy 1).** Matches user mental model ("I'm dragging horizontally, so I get a horizontal fence"). Implementation:

```typescript
// In handleMouseMove, new branch for edge painting:
if (isEdgePaintingRef.current) {
  const start = edgeStrokeStartRef.current!
  const dxTotal = p.rawGridX - start.x
  const dyTotal = p.rawGridY - start.y

  // Determine axis on first meaningful motion
  if (edgeStrokeAxisRef.current === null) {
    if (dxTotal === 0 && dyTotal === 0) return
    edgeStrokeAxisRef.current = Math.abs(dxTotal) >= Math.abs(dyTotal) ? 'h' : 'v'
  }

  const axis = edgeStrokeAxisRef.current
  // Project current point onto locked axis (ignore orthogonal component)
  const projected = axis === 'h'
    ? { x: p.rawGridX, y: start.y }
    : { x: start.x, y: p.rawGridY }

  // Rasterize edges from last sample to projected — each step of 1 along axis lays 1 edge
  const last = lastEdgePointRef.current!
  if (last.x === projected.x && last.y === projected.y) return

  const edges = rasterizeEdgeLine(last, projected, axis, fixture.id)
  // Skip first edge (last sample already committed)
  for (let i = 1; i < edges.length; i++) {
    placeEdgeIfAllowed(edges[i], fixture)
  }
  lastEdgePointRef.current = projected
  return
}
```

### Edge rasterization (pure function)

`rasterizeEdgeLine(from, to, axis, fixtureId)` produces a sequence of `PlacedEdge` objects along one axis between two lattice points:

```typescript
// src/utils/edgeRasterize.ts
export function rasterizeEdgeLine(
  from: { x: number; y: number },
  to: { x: number; y: number },
  axis: EdgeOrientation,
  fixtureId: number,
): Array<Omit<PlacedEdge, 'id'>> {
  const edges: Array<Omit<PlacedEdge, 'id'>> = []
  if (axis === 'h') {
    // Horizontal edges connect lattice points along x; y is fixed to from.y
    const y = from.y
    const x0 = Math.min(from.x, to.x)
    const x1 = Math.max(from.x, to.x)
    // N lattice-point segments → N horizontal edges at (x, y) for x ∈ [x0, x1-1]
    // (edge (x,y,'h') sits between tiles (x, y-1) and (x, y))
    // Actually: from x0 to x1 covers (x1 - x0) segments; we emit that many edges
    for (let x = x0; x < x1; x++) {
      edges.push({ fixtureId, x, y, orientation: 'h' })
    }
    // Degenerate single-point: no edge
  } else {
    const x = from.x
    const y0 = Math.min(from.y, to.y)
    const y1 = Math.max(from.y, to.y)
    for (let y = y0; y < y1; y++) {
      edges.push({ fixtureId, x, y, orientation: 'v' })
    }
  }
  return edges
}
```

The "skip first edge" trick from the tile path isn't needed here because edges sit **between** lattice points — a segment from lattice (2,3) to (4,3) produces edges at x=2 and x=3 (two edges, not three). The caller tracks `lastEdgePointRef` as the lattice endpoint already committed, so the next sample's rasterization naturally starts from that point.

### Dominant axis tie-break

When `|Δx| === |Δy|`, bias toward horizontal (`'h'`) — matches `snapToAxis` in `rasterize.ts:89` which uses `dx >= dy ? horizontal : vertical`. Consistent user experience.

## Edge-Tile Coexistence (Rendering + Hit-Testing)

### Render layering

Current layer order in `EditorCanvas.tsx` (bottom to top):
1. `GridLayer` (grass + dashed grid lines)
2. `GroundLayer` (roads, color tiles, rugs)
3. `FurnitureLayer`
4. `GhostPreview`
5. `FenceLineTool` (Phase 2 — being deleted)

**New layering:**
1. `GridLayer`
2. `GroundLayer` (roads, color tiles, rugs)
3. **`FenceLayer` (NEW)** — edges render above ground tiles, below furniture
4. `FurnitureLayer`
5. `GhostPreview` (tile-based brush ghost)
6. **Fence drag preview (NEW)** — a transient Konva `<Layer>` rendering the in-progress stroke's axis-locked edges in green (valid) / red (blocked). This replaces the deleted `FenceLineTool.tsx` ghost layer.

**Why fences above ground, below furniture:** In-game screenshots show fences are visually taller than road tiles but below furniture like chairs/tables sitting adjacent to them. Matching the game's visual depth-order preserves reproducibility (D-45).

### Fence rendering as thin rectangles

For an edge `{x, y, orientation: 'h'}`:

```typescript
<Rect
  x={x * TILE_SIZE}
  y={y * TILE_SIZE - FENCE_THICKNESS / 2}
  width={TILE_SIZE}
  height={FENCE_THICKNESS}
  fill={fenceColor}
/>
```

where `FENCE_THICKNESS ≈ 6` pixels (visible but not overpowering). Vertical edge swaps width/height. Fences don't need rotation, selection outlines, or drag handles in v1 (deferred — no fence selection UX yet; remove-mode drag-erase is the only edit operation).

### Hit testing

v1 scope: **no per-edge click selection.** Fences are place-and-forget; to remove, user enters remove mode and drags across them. Rationale: edge hit targets are 6px thin — precise clicking is frustrating and the game itself doesn't offer edge-level selection. The plan should explicitly document this as "remove-only" for v1 and defer selection to a future phase.

**Remove-mode drag-erase for edges:** Mirror the existing `eraseGroundTileIfAny` pattern — on each mouseMove sample in remove mode, iterate `placedEdges` and remove any whose `(x,y,orientation)` lies within a 1-tile radius of the cursor. Or simpler: snap cursor to nearest edge lattice point, compute `edgeKey`, remove if present.

**Tile-level operations coexist cleanly:** A road at tile (3,4) and a horizontal fence at (3,4,'h') occupy completely independent data structures. `buildOccupancyGrid` only looks at `placedItems`; `buildEdgeOccupancySet` only at `placedEdges`. No cross-layer collision check needed — layer independence (D-41) extends to the edge/tile boundary.

## Architecture Patterns

### Pattern 1: Parallel occupancy structures (tile + edge)

```typescript
// Two independent maps, two independent check functions
buildOccupancyGrid(placedItems, fixtureMap, 'ground') → Map<tileKey, itemId>
buildEdgeOccupancySet(placedEdges)                    → Set<edgeKey>

// In handleStageMouseDown, dispatch based on interaction type:
if (interaction === 'drag-paint')      → use tile occupancy
if (interaction === 'drag-paint-edge') → use edge occupancy
```

**When to use:** Two orthogonal entity types that must coexist at the same coordinate without collision. Edges and tiles are the canonical case. **Do not** try to shoehorn edges into `placedItems` with a synthetic `layer='edge'` — that forces every tile-occupancy consumer to filter out edges, spreading the complexity.

### Pattern 2: Ref-based drag state machines (existing — extend, don't rewrite)

Phase 2 Wave 3 established the pattern of using `useRef` for drag-in-progress state (`isPaintingRef`, `lastPaintedTileRef`, `isErasingRef`). The reason is documented in `EditorCanvas.tsx:161`: per-sample `setState` would trigger full canvas re-renders at 60fps, killing performance. This phase **adds a third ref-pair** (`isEdgePaintingRef`, `lastEdgePointRef`, `edgeStrokeAxisRef`) following the same pattern. The R-06 mouseleave/blur cleanup via `endStrokeIfActive` extends naturally.

**Anti-pattern:** Don't store `fenceState: {phase, start, end}` in `useState` as Phase 2 did. That was necessary for click-line-tool (UI needed to react to phase changes), but for drag-paint the entire in-progress state lives in refs and commits to the store on mouseup.

### Pattern 3: Discriminated routing at one choke point

`getBrushInteraction(fixture)` is the **single** place that decides "tile vs edge" for the drag path. The type system (discriminated union `'drag-paint' | 'drag-paint-edge' | null`) enforces exhaustiveness. This keeps the dispatch visible and greppable — future developers can find every tile/edge branch by searching for `getBrushInteraction`.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Batched undo for multi-edge strokes | Custom temporal snapshot logic | `withBatchedUndo` / `startStrokeBatch`+`endStrokeBatch` in `editorStore.ts:366` | Already battle-tested for brushStroke and brushErase; handles R-06 mouseleave/blur race correctly; idempotent via `_preStrokeSnapshot` guard |
| Line rasterization between lattice points | Custom Bresenham | For tile paths: reuse `rasterizeLine` from `utils/rasterize.ts`. For edge paths: dedicated `rasterizeEdgeLine` helper (simpler — always single-axis after axis-lock, so just a 1D loop, not Bresenham) | Single-axis edge lines don't need Bresenham at all |
| Drag event wiring on `<Stage>` | Custom pointer-event listeners | Konva's `onPointerDown`/`onPointerMove`/`onPointerUp` on `<Stage>` (already wired at EditorCanvas L504-509) | Free event normalization; existing handlers already consume these |
| Grid → pixel coordinate math | Recompute in each component | `TILE_SIZE` constant from `utils/grid.ts` multiplied directly | Already the established pattern in every render component |

**Key insight:** Phase 2's brush infrastructure was designed well enough that Phase 02.1 is mostly *deletion + minor extension*. Resist the urge to rewrite the stroke batching, rasterization, or pointer handling — they already work.

## Common Pitfalls

### Pitfall 1: Edge coordinates at lattice vs tile confusion

**What goes wrong:** Developer writes `rasterizeEdgeLine` assuming edge `(x,y)` is a tile, produces N+1 edges for an N-tile line, or off-by-one on boundary.

**Why it happens:** Tile coordinates (0..W-1) and lattice coordinates (0..W) differ by one. Horizontal edges have `x ∈ [0, W)` but `y ∈ [0, W]` — different ranges per axis per orientation.

**How to avoid:** Write bounds table in a comment at the top of `edgeRasterize.ts` (copy from "Coordinate semantics" section above). Add an explicit test: "horizontal edges at y=0 and y=D are valid (top and bottom borders); horizontal edge at y=D+1 is rejected."

**Warning signs:** Tests that place edges and get `N+1` when expecting `N`. Boundary fences disappearing at grid edges.

### Pitfall 2: Forgetting to extend zundo partialize

**What goes wrong:** Edges place correctly during a stroke, but undo doesn't restore them — the temporal middleware only snapshots `placedItems`.

**Why it happens:** `editorStore.ts:269` has `partialize: (state) => ({ placedItems: state.placedItems })`. The partialize selector is called at every tracked state change; fields not included are invisible to zundo.

**How to avoid:** Update `partialize` to include `placedEdges`. Update the `equality` comparator to check both references. Add a test: "undo after edge stroke restores empty `placedEdges`" (already in the test plan above).

**Warning signs:** Undo test fails while placement works.

### Pitfall 3: Axis-lock flicker from decision-per-sample

**What goes wrong:** User drags slightly off-axis; each mouseMove sample re-decides the axis and the ghost preview flickers horizontal↔vertical.

**Why it happens:** Naive "recompute dominant axis every frame" strategy.

**How to avoid:** Lock on **first cross-tile sample** and store in `edgeStrokeAxisRef`. Once set, never recompute within the stroke. Reset on mouseup (which clears all stroke refs via `endStrokeIfActive`).

**Warning signs:** Users report "fence jumps between horizontal and vertical when I drag diagonally."

### Pitfall 4: Stage drag panning vs brush drag conflict

**What goes wrong:** Edge brush stroke triggers Stage pan instead of painting.

**Why it happens:** `<Stage draggable={toolMode === 'select'}>` at L499. For brush mode this is already `false`, so no conflict. **But** if the plan ever adds brush-mode Stage dragging (e.g., space-bar pan override), that would break this.

**How to avoid:** Don't change the `draggable` prop. The current logic is correct; preserve it.

**Warning signs:** Mouse-down on a fence stroke initiates pan.

### Pitfall 5: Edge erase iterating all edges every sample

**What goes wrong:** Remove-mode drag-erase calls `Object.values(placedEdges).filter(...)` on every mouseMove sample. At 60fps with 200 edges, that's 12k iterations/sec — fine, but if it grows to 2000 edges it might stutter.

**How to avoid:** For v1, iterate directly — it's simple and fast enough. If profiling shows hotspot, build a transient occupancy Set inside `endStrokeIfActive` or memoize per-stroke. Do not prematurely optimize.

**Warning signs:** Profiler shows `eraseEdgeAt` as hot; don't add complexity until it does.

### Pitfall 6: Missing `placedEdges` in `startEditor` initial state

**What goes wrong:** `useEditorStore.getState().placedEdges` is `undefined` on app start; first edge placement throws.

**How to avoid:** Initialize `placedEdges: {}` in the Zustand store creator (alongside `placedItems: {}` at L109). Initialize it in `startEditor` reset too (L229-252) — preserve empty edges when switching area levels.

**Warning signs:** `TypeError: Cannot read properties of undefined (reading 'xyz')` on first fence placement after a fresh load.

```typescript
placedEdges: {} as Record<string, PlacedEdge>,

placeEdge: (edge: Omit<PlacedEdge, 'id'>) =>
  set((state) => {

    return { placedEdges: { ...state.placedEdges, [id]: { ...edge, id } } }
```

### Extended handleMouseMove edge branch (EditorCanvas.tsx)

```typescript
// Place after the existing isPaintingRef branch, before isErasingRef branch
if (isEdgePaintingRef.current) {
  const state = useEditorStore.getState()
  const fixture = state.activeFixtureId !== null
    ? fixtureMap.get(state.activeFixtureId) : undefined
  if (!fixture) return

  const start = edgeStrokeStartRef.current!
  const dxTotal = p.rawGridX - start.x
  const dyTotal = p.rawGridY - start.y

  if (edgeStrokeAxisRef.current === null) {
    if (dxTotal === 0 && dyTotal === 0) return
    edgeStrokeAxisRef.current = Math.abs(dxTotal) >= Math.abs(dyTotal) ? 'h' : 'v'
  }

  const axis = edgeStrokeAxisRef.current
  const projected = axis === 'h'
    ? { x: p.rawGridX, y: start.y }
    : { x: start.x, y: p.rawGridY }

  const last = lastEdgePointRef.current!
  if (last.x === projected.x && last.y === projected.y) return

  const edgeSet = buildEdgeOccupancySet(state.placedEdges)
  const newEdges = rasterizeEdgeLine(last, projected, axis, fixture.id)
  for (const edge of newEdges) {
    if (checkCanPlaceEdge(edgeSet, edge.x, edge.y, edge.orientation, state.gridSize.width, state.gridSize.depth)) {
      state.placeEdge(edge)
      edgeSet.add(edgeKey(edge.x, edge.y, edge.orientation))
    }
    // overwrite=OFF + occupied → skip (matches tile brush semantics)
  }
  lastEdgePointRef.current = projected
  return
}
```

## Open Questions

### 1. Should fence fixture colorCode drive `fenceColor`?

- **What we know:** Fixtures have a `colorCode` field; roads/color-tiles use `getFixtureColor(mainGenreId, colorCode)` in `PlacedItem.tsx:124`. v1 fences all belong to mainGenreId=13 and may have distinct colorCodes for the 7 variants.
- **What's unclear:** Whether fence variants should render in distinct colors or all wood-brown in v1.
- **Recommendation:** Reuse `getFixtureColor(fixture.mainGenreId, fixture.colorCode)` so variants are visually distinct. One line change from the example above. Planner should verify there are no color collisions with road mainGenreId=12.

### 2. Remove-mode drag-erase scope — edges only, or edges + tiles in one pass?

- **What we know:** Current `eraseGroundTileIfAny` only removes `placedItems` on the ground layer.
- **What's unclear:** Should a remove-mode drag over a tile that has both a road and a fence on the same square remove both, or only the top-most?
- **Recommendation:** Remove both (top-most first = fence since it renders above). Iterate `placedEdges` first, then `placedItems`. Document in plan. Alternatively, hold Shift to switch which layer is targeted — but that's v2 polish.

### 3. Fence variant selection mid-stroke

- **What we know:** User picks a fence fixture from the catalog, all edges in that stroke use that fixtureId.
- **What's unclear:** Should consecutive strokes with different fence variants coexist at the same edge location? (Probably not — one edge, one fixture.)
- **Recommendation:** Disallow overlap regardless of variant. Use `checkCanPlaceEdge` rejection. If `overwriteEnabled`, replace via removeEdge + placeEdge in the same batch (matches tile overwrite semantics).

### 4. Deletion of `'line-tool'` enum value — breaking change or rename?

- **What we know:** `getBrushInteraction` is called at `EditorCanvas.tsx` (via useFenceLineTool deletion path), `GhostPreview.tsx:55`, `pickToolModeForFixture` in `editorStore.ts:47`.
- **What's unclear:** Whether to rename to `'drag-paint-edge'` or collapse to `'drag-paint'` with a separate `isEdgeBrush(fixture)` predicate.
- **Recommendation:** **Rename.** Keeps discriminated-union exhaustiveness checks working, makes the dispatch visible at every call site. Two renames: function return type + test assertion.

### 5. Does axis-lock need a "commit on mouseup if no cross-tile" case?

- **What we know:** A user clicks once without dragging — at mouseup, `edgeStrokeAxisRef` is still `null`.
- **What's unclear:** Should this commit a single edge at the clicked lattice point? Or be a no-op?
- **Recommendation:** **No-op.** A single click has no direction — we can't tell if the user meant a horizontal or vertical edge. The stroke batch ends with 0 edges placed, which is consistent with "needs drag to place fences." Document this in the plan's success criteria.

### 6. Edge preview during drag — how to render?

- **What we know:** The deleted `FenceLineTool.tsx` rendered a ghost Layer with semi-transparent rects. The current `GhostPreview.tsx` hides itself for edge fixtures (L60-62).
- **What's unclear:** Should the in-progress stroke render a live preview layer, or rely on edges being written to `placedEdges` and rendered by `FenceLayer` in real-time?
- **Recommendation:** **Rely on `FenceLayer` rendering committed edges in real-time.** The stroke writes edges to `placedEdges` as the mouse moves (via `state.placeEdge`); `FenceLayer` subscribes and renders them instantly. No separate preview layer needed. This is simpler than the tile brush (which uses `paintTileIfAllowed` with a separate ghost) because edges are committed progressively through the stroke and the temporal batch defers undo — so even "preview" edges are real edges that get collapsed into one undo step on mouseup. This is already how the tile brush works — no special preview layer.
