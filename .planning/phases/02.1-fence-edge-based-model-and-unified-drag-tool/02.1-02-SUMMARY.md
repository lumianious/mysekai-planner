## Files Created/Modified

- `src/utils/edgeRasterize.ts` — NEW. Five pure exports, L3 header documenting coordinate semantics, no runtime imports beyond types

## Decisions Made

- **Edge occupancy check location:** Kept occupancy helpers inside `edgeRasterize.ts` (as utilities consumed by the drag handler in Plan 03), not duplicated into the store. The store stays slim and testable.
- **StrokeSnapshot extension:** Rather than only extending `partialize`, also extended `_preStrokeSnapshot` to include `placedEdges`. Without this, Plan 03's fence strokes would undo items but leave edges stuck in post-stroke state — a latent correctness bug.
- **Inclusive-end bounds:** `checkCanPlaceEdge` follows research convention strictly: 'h' edges valid for `y<=D`, 'v' edges valid for `x<=W`. This matches the "edge on the boundary of the last tile" visual mental model.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing Critical] Extended `_preStrokeSnapshot` / `endStrokeBatch` to include `placedEdges`**
- **Found during:** Task 3 (zundo partialize extension)
- **Issue:** The plan's Task 3 action list explicitly said "Do NOT touch startStrokeBatch/endStrokeBatch". But the existing implementation captured only `placedItems` into `pastStates`. Because `partialize` now returns both fields, writing a snapshot with only `placedItems` would produce a past-state of the wrong shape — undoing a batched stroke would restore items but leave `placedEdges` unchanged (stuck at post-stroke state). Plan 03's unified fence drag stroke relies on `withBatchedUndo`, so this would silently break fence undo.
- **Fix:** Introduced `StrokeSnapshot` interface `{ placedItems, placedEdges }`. `startStrokeBatch` now captures both; `endStrokeBatch` writes the full snapshot object into `pastStates`. Behavior for item-only strokes is identical (the new `placedEdges` field in the snapshot equals the current edges, so undo is a no-op for edges). Behavior for edge-only and mixed strokes is now correct.
- **Files modified:** `src/stores/editorStore.ts`
- **Verification:** `pnpm test --run src/__tests__/editorStore.test.ts src/__tests__/temporalBatch.test.ts` — 38/38 passing (6 temporalBatch tests still green, confirming no regression on the existing item-only batching path)
- **Committed in:** `b68514b` (Task 3 commit)
