# Phase 02.1 Plan 05: Bidirectional Fence-Furniture Collision Summary

**Fence edges and furniture now block each other's placement ‚Äî both are level 1 items that cannot overlap.**

## Performance

- **Duration:** ~8 min
- **Tasks:** 2
- **Files modified:** 4 (0 created, 4 edited)

## Accomplishments

- New `getEdgesForTileFootprint(x, y, w, d)` in edgeRasterize.ts ‚Äî returns all edge keys bordering a rectangular footprint
- `checkCanPlaceEdge` extended with optional `occupancyGrid` param ‚Äî blocks edges adjacent to furniture tiles
- `checkCanPlace` extended with optional `edgeSet` param ‚Äî blocks furniture whose footprint borders existing edges
- `paintEdgeStroke` in EditorCanvas builds furniture OccupancyGrid and passes to checkCanPlaceEdge
- `handleCanvasClick` and `handleMoveItem` in useEditorActions build edge sets and pass to checkCanPlace
- `paintTileIfAllowed` deliberately NOT wired ‚Äî roads coexist with fences by design

## Task Commits

1. **Task 1: Cross-entity collision helpers** ‚Äî `60765fd` (feat)
2. **Task 2: Wire into all placement paths** ‚Äî `5dcea06` (feat)

## Decisions Made

- **Optional params over overloads:** Simplest extension. All existing call sites (no new param) behave identically ‚Äî zero risk of regression.
- **Roads excluded:** `paintTileIfAllowed` (ground-layer tiles) does NOT check edge collision. Fences and roads coexist by game design ‚Äî UAT test 4 confirmed this.
- **Type-only import:** edgeRasterize.ts uses `import type { OccupancyGrid }` from editorStore ‚Äî erased at runtime, no circular dependency.
